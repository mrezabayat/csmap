## In simple terms

You should never store the password the user typed. Instead, run it through a **password hashing function** — a one-way, deliberately slow algorithm with a random per-user **salt** — and store the result. To log in, hash the new attempt and compare. A leaked database is then a list of guesses, not a list of passwords.

## More detail

A password hash is not a general hash. SHA-256 is too fast; an attacker with a GPU can try billions per second. The right algorithms are designed to be expensive in CPU time *and* memory, with **tunable cost**:

| Algorithm  | Notes                                                           |
|------------|-----------------------------------------------------------------|
| **Argon2id** | OWASP's first recommendation in 2026. Winner of the PHC contest. |
| **scrypt**   | Memory-hard, widely supported.                                  |
| **bcrypt**   | Older, still acceptable, 72-byte input limit, no memory hardness. |
| **PBKDF2**   | FIPS-approved; use only when policy requires it.                |
| **SHA-256 / MD5** | **Not password hashes.** Never use them for this.          |

Each user gets a unique random **salt** (16+ bytes) stored alongside the hash. This means identical passwords produce different stored values, defeating rainbow tables.

Many systems also add a **pepper** — a secret kept outside the database (e.g. in a KMS) and applied via HMAC before hashing. A pure DB leak then yields nothing.

The cost parameter is tuned to take ~250–1000 ms on your hardware. Increase it over time as hardware gets faster.

When checking a password:

- Always use a **constant-time** comparison on the resulting hash to avoid timing attacks.
- Rate-limit login attempts.
- Detect known-breached passwords (e.g. Have I Been Pwned's k-anonymity API) and refuse them.

## Why it matters

Almost every "we hashed your passwords" reassurance after a breach hinges on *which* hash. With Argon2id and a per-user salt, the leak is largely useless to the attacker. With a single round of SHA-1, the entire user base is compromised within hours.

## Real-world examples

- The 2012 LinkedIn breach: 6.5 million SHA-1 hashes, no salt. Most cracked within days.
- The 2013 Adobe breach: passwords encrypted with a single key (not hashed), letting attackers correlate identical passwords.
- Modern frameworks (Django, Rails, Laravel, ASP.NET Core) default to Argon2id or bcrypt out of the box.

- The 2012 LinkedIn breach exposed 6.5 million SHA-1 hashes; >90% were cracked within days. The same data, hashed with Argon2id, would have been worthless to attackers.

## Common misconceptions

- **"Hashing means encrypting."** Encryption is reversible with the key; hashing is one-way.
- **"A complex password makes hashing unnecessary."** Slow hashing buys time even when passwords are weak; it is layered protection.
