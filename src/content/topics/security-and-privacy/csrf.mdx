## In simple terms

**Cross-Site Request Forgery (CSRF)** exploits the fact that browsers automatically attach cookies to requests. The victim is logged into `bank.com` in one tab; in another tab they visit `attacker.com`, which contains an invisible form that POSTs to `bank.com/transfer?to=attacker&amount=10000`. The browser dutifully attaches the bank's session cookie. The bank sees an authenticated request and processes the transfer.

## More detail

The classic CSRF setup:

1. Victim is signed into `bank.com`.
2. Attacker emails them a link to `evil.com`.
3. `evil.com` contains:
   ```html
   <form id="x" action="https://bank.com/transfer" method="POST">
     <input name="to" value="attacker">
     <input name="amount" value="10000">
   </form>
   <script>document.getElementById('x').submit();</script>
   ```
4. Victim visits `evil.com`; browser submits the form with the bank's cookies.

Why it works: the browser attaches cookies to *any* request to a domain, regardless of which page initiated it.

Defences (use in combination):

1. **`SameSite` cookies** — `SameSite=Lax` blocks cookies on cross-site POSTs and most cross-site requests; `SameSite=Strict` blocks them on cross-site navigation too. Browsers default to `Lax` since ~2020, which prevented many naive CSRFs overnight. Set it explicitly.
2. **CSRF tokens** — server includes a random per-session token in every form; legitimate POSTs include it; cross-site forms can't read it (same-origin policy). Most server frameworks (Django, Rails, Laravel, Spring) do this by default.
3. **Custom request headers** — JavaScript-driven APIs require `X-Requested-With: XMLHttpRequest` or similar; cross-site forms can't set custom headers (without explicit CORS approval).
4. **Origin / Referer checks** — verify the request came from your own domain.
5. **Re-authentication for sensitive actions** — money transfers, password changes, mfa changes — re-prompt for password regardless of session.

**SameSite=Lax becoming the browser default** has dramatically reduced naive CSRF, but it hasn't killed it: same-site subdomain attacks still work (`a.example.com` vs `b.example.com`), and Lax still allows top-level GET navigation.

CSRF is conceptually about **forcing a request**; XSS is about **executing code inside the origin**. XSS is strictly worse (it can do everything CSRF can plus much more) — if you have XSS, CSRF defences don't help.

## Why it matters

CSRF caused some of the most embarrassing web bugs of the 2000s and early 2010s. Modern browsers and frameworks have made the basic attack much harder, but the bug class isn't dead — APIs, mobile webviews, and unusual cookie configurations still produce CSRF vulnerabilities every year.

## Real-world examples

- **The classic Gmail CSRF (2007)** changed account recovery email via a forged form — fixed and disclosed by Robert Graham, leading to widespread CSRF awareness.
- **A 2018 Twitter CSRF** let attackers tweet on behalf of any logged-in user via a crafted link; fixed within days.
- The shift to **`SameSite=Lax` as default** in Chrome 80 (2020) measurably reduced CSRF in the wild — and broke some legitimate cross-site integrations, prompting a wave of `SameSite=None; Secure` cookie reconfigurations.
- Django's `csrf_token` template tag — automatically included via middleware — has prevented countless CSRFs since 2005.

## Common misconceptions

- **"HTTPS prevents CSRF."** HTTPS protects in-flight content; CSRF is about a request the user's browser issued willingly. Orthogonal.
- **"GET requests are safe."** They aren't if they change state. Make sure GET is read-only; use POST / PUT / DELETE for changes.
